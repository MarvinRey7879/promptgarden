# Deep Research #3 — AI Coding Benchmark Landscape (for promptgarden "Benchmarks" pillar)

Research date: 2026-07-11. All URLs live-checked (HTTP 200) on this date unless noted. Numbers change weekly on these leaderboards — treat every score below as a snapshot, not a fact that ages well.

## Executive summary

- The benchmark landscape has fragmented past SWE-bench: agentic/terminal benchmarks (Terminal-Bench, Aider Polyglot), continuously-refreshed contamination-resistant ones (LiveCodeBench), human-preference arenas (LMArena/WebDev Arena), and composite "index" scores (Artificial Analysis Coding Agent Index) now coexist, each measuring a different slice of "can this model code" ([swebench.com](https://www.swebench.com/), [tbench.ai](https://www.tbench.ai/), [aider.chat](https://aider.chat/docs/leaderboards/), [livecodebench.github.io](https://livecodebench.github.io/), [arena.ai](https://arena.ai/leaderboard/code), [artificialanalysis.ai](https://artificialanalysis.ai/agents/coding-agents)).
- Two "elder" benchmarks are effectively retired for frontier-model comparison: HumanEval and MMLU are both saturated (90%+ scores, narrow differentiation) and now function only as minimum-bar sanity checks, not rankings ([benchmarkingagents.com](https://benchmarkingagents.com/humaneval/), [tokenmix.ai](https://tokenmix.ai/blog/mmlu-benchmark-leaderboard)).
- Even a single benchmark can yield contradictory "official" top scores depending on agent scaffolding and which data split is used — SWE-bench Pro alone has at least three defensible "best score" claims in circulation, which is the single biggest trap for a non-expert reader ([morphllm.com](https://www.morphllm.com/swe-bench-pro), confirmed independently against Scale's own leaderboard: [labs.scale.com](https://labs.scale.com/leaderboard/swe_bench_pro_public)).
- Structural criticism is well documented and specific, not vague grumbling: SWE-bench has measured issues like solution leakage in the underlying issue threads and weak test cases that let broken fixes pass ([runloop.ai](https://runloop.ai/blog/swe-bench-deep-dive-unmasking-the-limitations-of-a-popular-benchmark)), and vendors are documented cherry-picking their best variant/prompt for publication ([ucstrategies.com](https://ucstrategies.com/news/ai-benchmarks-are-a-game-now-and-the-industry-is-cheating-to-win/)).
- Given how fast numbers rot (one official leaderboard I fetched today, aider.chat's, still shows GPT-5/o3-pro-era models on top — not because those are current SOTA, but because the leaderboard hasn't been refreshed), promptgarden should not embed live-looking score tables; a methodology-first explainer + per-benchmark cards linking out to official sources is the lower-maintenance, more trustworthy design (see Recommendation section).

## 1. The benchmarks that matter in 2026

- **SWE-bench (family: original, Verified, Multilingual, Multimodal, Lite, Pro)** — measures whether an AI agent can resolve real, closed GitHub issues from open-source repos by producing a patch that passes the repo's hidden test suite. Built and maintained by researchers from Princeton and Stanford (original authors: Yang, Jimenez, et al.); the human-filtered "Verified" subset (500 instances) was produced with OpenAI; the harder "Pro" variant (1,865 tasks, 41 repos, contamination-resistant) is run separately by Scale AI. Official: [swebench.com](https://www.swebench.com/) (main), [swebench.com/verified.html](https://www.swebench.com/verified.html) (Verified), [labs.scale.com/leaderboard/swe_bench_pro_public](https://labs.scale.com/leaderboard/swe_bench_pro_public) (Pro, Scale AI) — [princeton-nlp/SWE-bench on Hugging Face](https://huggingface.co/datasets/princeton-nlp/SWE-bench) confirms Princeton NLP as the dataset owner.
- **Terminal-Bench (2.0 / 2.1)** — measures whether an agent can complete real, multi-step command-line tasks (sysadmin, debugging, training a small model, compiling code) where each command's output changes the next decision. A Stanford x Laude (Anthropic-adjacent) collaboration, part of the open "Harbor" framework. Official: [tbench.ai](https://www.tbench.ai/).
- **Aider Polyglot** — measures whether a model can follow instructions and edit code correctly, unassisted, across 225 Exercism exercises in 6 languages (C++, Go, Java, JavaScript, Python, Rust), scored as pass-rate after up to 2 attempts with a "well-formed edit" sub-score. Maintained by Paul Gauthier (Aider's creator), not a lab or consortium. Official: [aider.chat/docs/leaderboards](https://aider.chat/docs/leaderboards/).
- **LiveCodeBench** — measures code generation, self-repair, test-output prediction and code execution using competitive-programming problems pulled continuously from LeetCode, AtCoder and Codeforces, each timestamped so evaluators can test only on problems released *after* a model's training cutoff (its core anti-contamination design). Maintained by researchers at UC Berkeley, MIT and Cornell (lead: Naman Jain). Official: [livecodebench.github.io](https://livecodebench.github.io/).
- **LMArena Code Arena / WebDev Arena** — measures human preference, not correctness: real users blind-vote on which of two models' code/UI output they prefer, aggregated into an Elo-style score; WebDev Arena specifically scores front-end/web-app generation including agentic multi-step workflows. Run by Arena.ai (the LMArena/LMSYS project, now on the arena.ai domain — note the old lmarena.ai URL now 301-redirects here). Official: [arena.ai/leaderboard/code](https://arena.ai/leaderboard/code) and [arena.ai/leaderboard/code/webdev](https://arena.ai/leaderboard/code/webdev).
- **HumanEval** — measures pass@k functional correctness on 164 hand-written Python programming problems by actually executing generated code against unit tests. Created by OpenAI (Chen, Tworek, Jun et al., 2021). Official: [github.com/openai/human-eval](https://github.com/openai/human-eval). Status: **saturated/legacy** — frontier models cluster at 93–98%, so it's used only as a floor/sanity check in 2026, not for ranking frontier models ([benchmarkingagents.com](https://benchmarkingagents.com/humaneval/)).
- **Artificial Analysis Coding Agent Index** — a composite index measuring end-to-end coding-*agent* performance (not just the underlying model), averaging pass@1 across three component benchmarks (DeepSWE, Terminal-Bench v2, SWE-Atlas-QnA) plus pooled cost/token/time efficiency metrics; explicitly designed to score the model+harness pairing (e.g. "GPT-5.6 in Codex" vs "GPT-5.6 in some other harness") rather than the model alone. Run by Artificial Analysis, an independent benchmarking outfit. Official: [artificialanalysis.ai/agents/coding-agents](https://artificialanalysis.ai/agents/coding-agents), methodology at [artificialanalysis.ai/methodology/coding-agents-benchmarking](https://artificialanalysis.ai/methodology/coding-agents-benchmarking). This is the "Coding Agent Index" referenced in GPT-5.6 coverage.
- **ARC-AGI-2** — not a coding benchmark per se; measures abstract/fluid reasoning (symbolic interpretation, compositional and contextual rule application) via visual grid-transformation puzzles, plus an efficiency dimension (cost to solve, not just accuracy). Included here as general reasoning context since coding-agent capability claims increasingly cite it alongside coding scores. Run by the nonprofit ARC Prize Inc. (private/semi-private test splits to resist contamination). Official: [arcprize.org/leaderboard](https://arcprize.org/leaderboard), benchmark description at [arcprize.org/arc-agi/2](https://arcprize.org/arc-agi/2).
- **MMLU / MMLU-Pro** — general knowledge/reasoning multiple-choice benchmark (57 subjects originally; MMLU-Pro expands to ~12,000 questions across 14 disciplines with 10 answer choices to cut guessing). Not coding-specific; included as general context per your candidate list. Origin: Hendrycks et al., UC Berkeley-affiliated (2020 paper). Reference: [arxiv.org/abs/2009.03300](https://arxiv.org/abs/2009.03300), dataset page [paperswithcode.com/dataset/mmlu](https://paperswithcode.com/dataset/mmlu). Status: **saturated** — frontier models score 89–92%, too narrow a band to differentiate; MMLU-Pro is the current standard for this axis, and even it is reported to be starting to saturate in early 2026 ([tokenmix.ai](https://tokenmix.ai/blog/mmlu-benchmark-leaderboard), [presenc.ai](https://presenc.ai/research/mmlu-pro-leaderboard-june-2026)).
- **METR "Time Horizon" benchmark** *(new/trending addition beyond your candidate list)* — measures the length of task (in human-expert-hours) an agent can complete autonomously at a fixed reliability threshold, tracked longitudinally since 2019 across ~230 mostly-coding tasks; the widely-cited finding is that this horizon has been doubling roughly every 7 months. This has become a standard reference point in coding-agent capability discourse precisely because it's a trend line rather than a single leaderboard snapshot. Run by METR, an independent AI evaluation nonprofit. Official: [metr.org/time-horizons](https://metr.org/time-horizons/). Caveat noted by METR itself: measurements above ~16 hours are flagged as unreliable with their current task suite.

*(Also surfaced as emerging/worth watching but not detailed above: SWE-rebench, a continuously-mined GitHub-issue benchmark built specifically to fight contamination by timestamping task creation against model release dates — [swe-rebench.com](https://swe-rebench.com/).)*

## 2. How to read benchmark numbers — known criticisms

- **Contamination is structural, not occasional.** Public benchmark items propagate into web-scraped pretraining corpora (Common Crawl, GitHub, Reddit) over time; MMLU questions have been found verbatim in Common Crawl and HumanEval problems echo public LeetCode solutions, so a model can score well by memorization rather than capability ([llm-stats.com](https://llm-stats.com/blog/research/what-is-a-contaminated-llm)).
- **Saturation quietly kills a benchmark's usefulness before anyone retires it.** MMLU and HumanEval both saturated years ago (frontier models now scoring 90%+), which compresses all frontier models into a statistically meaningless few-point band — the benchmark keeps reporting numbers, they just stop meaning anything ([tokenmix.ai](https://tokenmix.ai/blog/llm-leaderboard-2026), [benchmarkingagents.com](https://benchmarkingagents.com/humaneval/)).
- **"Solvable" benchmark design has its own leakage problems.** An independent audit of SWE-bench found roughly a third of issues (32.67%) had their solution effectively given away in the issue thread/comments, another ~31% had test cases too weak to catch a broken fix, and of the fixes top models produced, 63.75% were flagged as "suspicious" (either leaked or genuinely wrong but test-passing) ([runloop.ai](https://runloop.ai/blog/swe-bench-deep-dive-unmasking-the-limitations-of-a-popular-benchmark)).
- **"Verified" reduces noise, it doesn't remove domain narrowness.** SWE-bench Verified is a human-filtered 500-instance subset curated with OpenAI to ensure tasks are clear and solvable — a real methodological improvement — but it inherits SWE-bench's underlying bias toward bug-fixing in popular open-source Python repos, not the full space of software engineering work ([swebench.com/verified.html](https://www.swebench.com/verified.html)).
- **A single benchmark name can hide multiple incompatible "official" scores.** SWE-bench Pro alone has circulating claims of ~59% (Scale's standardized public-set scaffolding), ~69% (a vendor-aggregate methodology), and ~47% (Scale's private commercial split) — all "real" depending on scaffold/data-split choice, which makes a bare percentage close to meaningless without knowing which variant produced it (reported by [morphllm.com](https://www.morphllm.com/swe-bench-pro); the ~59% figure for the public set was independently confirmed against Scale's own live leaderboard at [labs.scale.com](https://labs.scale.com/leaderboard/swe_bench_pro_public)).
- **Vendor self-reporting and cherry-picking are documented, not hypothetical.** Labs have been shown submitting only their best-performing variant/prompt configuration to leaderboards; one analysis of ~2.8M LMArena comparison records found selective submissions inflating scores by up to 100 Elo points, and one widely-cited case (Kimi K2 on HLE) showed a 50% claimed score vs. 29.4% independently measured ([ucstrategies.com](https://ucstrategies.com/news/ai-benchmarks-are-a-game-now-and-the-industry-is-cheating-to-win/)).
- **Public benchmark scores routinely overstate real-world task performance.** Teams running domain-specific evaluations (legal, medical, financial tasks) report accuracy 15–30% lower than the closest public benchmark would have predicted — the gap between "solves a curated task" and "solves your actual workflow" is large and consistent ([digiterialabs.com](https://digiterialabs.com/ai/insights/model-benchmarks-reality-check)).
- **Leaderboards go stale silently.** Not every maintainer re-runs every new flagship model — e.g. Aider's own polyglot leaderboard, fetched live today, still shows GPT-5/o3-pro/Gemini-2.5-Pro-era entries at the top rather than any 2026 flagship, because it simply hasn't been re-run against those models yet. A "current #1" badge can be many months out of date on a benchmark's own official page ([aider.chat/docs/leaderboards](https://aider.chat/docs/leaderboards/), observed directly 2026-07-11).

## 4. Recommendation — how promptgarden should present benchmarks

- **Don't embed a live-looking score table in written content.** Numbers here move on a timescale of weeks, not months (SWE-bench Pro alone has 3 incompatible "official" figures depending on scaffold/split; the Aider Polyglot leaderboard sat un-refreshed through at least one major model generation). Any static number promptgarden writes will read as stale or wrong within a short window, which damages trust in the whole pillar, not just that one figure.
- **Split into two content types matching their actual rate of change**: (1) one evergreen "how to read a benchmark" explainer entry covering the Section 2 criticisms (contamination, saturation, scaffold-dependent scores, vendor cherry-picking) — this content barely changes year over year and is the highest-leverage piece to get right once; (2) short per-benchmark mini-cards (name, one-sentence "what it measures," who runs it, official link) that stay accurate because they describe *methodology*, not *current standings*.
- **Every mini-card should link out to the official live leaderboard rather than quoting a score.** This offloads the freshness problem to the benchmark owner (who has the infrastructure to keep it current) instead of promptgarden manually chasing 10 leaderboards. It also sidesteps the "which of 3 official numbers do I quote" problem documented above.
- **If a "current results" feel is wanted anyway, timestamp everything hard.** Any place a number does appear (e.g. in an illustrative screenshot or a dated blog-style post, not the evergreen reference entry) should carry an explicit "as of [date], per [official source link]" — never an unlabeled bare percentage — so it ages as a dated snapshot rather than a false claim of currency.
- **Budget for a quarterly review pass, not continuous maintenance.** Given the benchmark landscape itself is churning (new variants like SWE-bench Pro, new entrants like the Artificial Analysis Coding Agent Index and METR's time-horizon metric appearing within the last 6–12 months), the mini-card set itself — not just the numbers inside it — needs a periodic "is this still the right list of 8-12 benchmarks" check, separate from and lower-frequency than any score-freshness concern.
